Test Report

Component Testing

Database

  When we implemented the habit and user tables in our main database, we had to test how they were created,
 edited, used and linked to one another. For the user, we tested creating multiple people with the same
  username, incorrect login credentials, and invalid username and password options. Multiple users were linked
 to their own habits. We made sure that habits were created within a user’s account and could be edited separately. The
 information we stored in the main database was tested for persistence in the local data of a user’s device.
 To see all of our changes - we gave each habit and user had unique ids that identified and linked them
 to one another. We were able to see in real time the changes we made in the database through the command line.
 In testing the database, we were able to test the key functions that are a part of the login and registration
 pages of our app.

Notifications

  To test notifications, we first tested whether or not a notification button would show up on our app. Then
 we scheduled notifications - some to occur once, at a specific time on a specific day, and others to occur
 epeatedly at a specific time each day. We made sure to test if we had a notification set for midnight, the
 notification would occur on the correct day. Another test we ran on this module was if the notification data
 was persistent & would sustain changes if a user decided to change the schedule. We also made sure that if
 the app was not currently open on the phone, we would still get notifications.

HomePage / HabitList

  Initially we did not have the database running, therefore to test the creation of a list of tiles for each
 habit, we created a local list with pre-made habits and their individual characteristics. We then gave the
 homepage access to the list and for each habit in the list a tile was printed to the screen with the name
 of the habit printed on the tile. The next step was to add more functionality to the tile. We wanted the user
 to be able to click a checkbox and be set as done or not done. This required us to change the state of a tile
 to change from done to not done or vice versa. Once the checkbox state changing was working, we were able to
 start incrementing and decrementing the streak count for each habit depending on the tiles done or not done
 state. At this early point the basics of the Habit List/tile creation was done.
  When we got the database up and running with a table that could hold the habits for an individual user,
 and those habits characteristics, we had to change how the system accessed those habits, and created a list
 from them.We first tested whether we could access the habits in the table, to do this we had to make sure
 we could add habits into the table. After adding habits to the database, we needed to initialize the database
 in the homepage so that the habits in the table could be accessed. Then for each habit in the database that is
 linked to the user that is logged in, the homepage prints a tile.

HabitTiles

  Initially the habit tile was very basic, it was just a box with the name of a habit printed in it.
 When we could get the box printed to the screen using the initial list of habits, we started working on adding
 a checkbox function to it to set a value for done or not done.
  Once the app was using the database to save and access habits, we started adding more functionality
 to the tile. Since we created a specific table for habits with individual characteristics and those values
 would be saved and accessible, we created an overlay that would come up when a tile is clicked. At this point
 we just tested to see if the overlay could be seen on screen, we then added a text box to the overlay to edit
 the name of a habit. To test this, we tested if we could write in the newly created textbox and then we checked
 the database to see if the name of the habit would change in the db. Next we added drop down boxes for
 milestones for users to decide at what point they get an award. We then tested whether the numbers chosen
 would be saved. At this point they are only saved locally in the overlay, but because our milestone table
 is not fully functional the data is not persistent yet.

Navigation

  For navigation, there are two different ways that the app handles navigation. There is navigation that
 is handled based on the states of certain pages like the loginpage, and then there is navigation using the
 navigation bar. To test the navigation done through the loginpage, we had to make sure that we could login
 successfully, if a user logged in successfully then the system would open the homepage. If a user has not
 been entered into the database yet, then can click a button on the loginpage that sends them to the registration
 page. These pages are only accessed internally upon opening the app. After making sure users could be added
 to the database, then we tested whether the login page would come up on opening the app and that it would
 send us to the correct page if the login was successful. Next we had to test that once the homepage opened
 after logging in, there would be a navigation bar, where you could click a button which would access the
 page linked via an index to jump to a different page. We then tested whether those buttons sent us to the
 correct pages.

System Testing

 Throughout our development, we were forced to do system testing of each component as they were built. We used
 this strategy in order to keep a coherence to the UI as well as to make sure our functionalities were behaving
 appropriately.To test how our app was running, it had to frequently be rebuilt via a prebuilt build.gradle which
 which is an automation tool for handling tasks, building the app, and running tests which would tell us whether
 there were any breakpoints in our code. It was critical that when the database was built and functioning, every
 other component drew information from it and sent back accurate information to it. Each time a page or
 major functionality was added to a component, we would clear the database and rerun the app as a new user.
 We also made sure to implement error messages for the user to show when there is invalid input.
 This allowed us to build on the UI/UX of the app as it was being built and find any functionality
 errors in our code.

Acceptance Testing

  To perform acceptance testing, we had to look back at our user stories and from that find the criteria for
  the acceptance testing:

User Registration:

 The user would want to register for an account so that they are able to add and track their habits.

 Acceptance criteria:
  Users should be able to enter their email, username, and password in order to create a new account.
   After registration, the user should be able to log in using the details they entered (stored in a
  database)

Addition of Habits

 A user would like to be able to add an activity once they are logged in to their account so that they can
 manage it.

 Acceptance criteria:
  The user should be able to enter the name of the activity they are trying to add, and the frequency
  of tracking(i.e. Daily, weekly).
  After the habit has been added, the user should be able to see the activity on their account’s dashboard.

Tracking a Habit

 Once the habit has been added, the user would like to be able to track the progress of the activity so they
 can see progress over time

 Acceptance criteria:
  The user should be able to click on the activity on the dashboard, and this should show them the
  progress chart for that activity, (streak of each activity- how many times the activity has been completed
   over a given period of time).

User Interface

 A user wants the app to be easy to use (navigating through the app’s features) and visually appealing.

 Acceptance criteria:
  The app should have clear labels for buttons and menu, consistency in the typography and color
  schemes with a layout that is easy to navigate. The user should be able to perform any task within
  the app quickly and easily without encountering any confusion or frustrating elements.

Reminders

 The user wanted to easily be able to set notifications to be reminded to perform each task.

 Acceptance criteria:
  The app has the ability to set notifications and for them to either be repeated, or one time.
  The user has the ability to manually set notifications or accept the default notification time.
